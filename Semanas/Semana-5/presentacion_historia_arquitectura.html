Historia de los Patrones de Arquitectura

Del código espagueti a los ecosistemas distribuidos

1960s - 1970s El Caos y la Modularidad

En los inicios, el software era "Monolítico" y desordenado (Spaghetti Code). El nacimiento de la ingeniería de software en 1968 impulsó la necesidad de orden.

Programación Estructurada

Se introducen las funciones y procedimientos para romper el código en bloques lógicos, sentando las bases de la separación de intereses.

1979 El Origen del MVC

Trygve Reenskaug introduce el Model-View-Controller en Xerox PARC para Smalltalk-80. Su objetivo era resolver el problema de la interacción del usuario con conjuntos de datos complejos.

Impacto: Fue el primer patrón en separar formalmente los datos (Modelo) de la interfaz (Vista), usando un mediador (Controlador).

1990s - 2000s Arquitectura en Capas

Con el auge de las aplicaciones empresariales (J2EE, .NET), se estandariza la Arquitectura en N-Capas.

Arquitectura en Capas

Divide el sistema en: Presentación, Negocio y Datos. Es la arquitectura "por defecto" para la mayoría de los monolitos modernos por su simplicidad.

SOA (Service Oriented)

El primer intento serio de crear sistemas distribuidos mediante servicios que se comunican por protocolos pesados (SOAP/XML).

2005 - 2012 Aislamiento del Dominio

Alistair Cockburn presenta la Arquitectura Hexagonal (2005) y Robert C. Martin la Clean Architecture (2012).

Filosofía: El centro de la aplicación es la "Lógica de Negocio" pura. Todo lo demás (BD, UI, APIs externas) son solo detalles que se conectan mediante adaptadores. Esto permite testear la lógica sin necesidad de una base de datos real.

2014 - Presente Microservicios y Cloud

Gigantes como Netflix y Amazon popularizan los Microservicios. El monolito se rompe en cientos de piezas independientes.

Microservicios

Escalabilidad extrema e independencia tecnológica. Cada servicio es dueño de su propia base de datos.

Serverless / Event-Driven

Arquitecturas que solo se ejecutan cuando ocurre un evento, optimizando costos y recursos al máximo.

¿Cuál es la más usada hoy?

No existe una "mejor", sino una "adecuada" según el contexto:

Tipo de Aplicación Arquitectura Predominante Razón
CRUDs / Startups tempranas MVC / Capas Velocidad de desarrollo y simplicidad.
Sistemas Financieros / Core Hexagonal / Clean Alta testeabilidad y seguridad del negocio.
E-commerce / Redes Sociales Microservicios Necesidad de escalar partes específicas (ej: Pagos).
IoT / Notificaciones Event-Driven Procesamiento masivo de señales en tiempo real.
