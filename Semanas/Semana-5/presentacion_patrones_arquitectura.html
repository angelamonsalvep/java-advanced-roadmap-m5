Patrones de Arquitectura de Software

Estrategias de diseño para sistemas escalables y mantenibles

Evolución del MVC

1. Arquitectura en Capas (N-Tier)

Es la evolución natural del MVC. Divide la aplicación en capas horizontales donde cada una tiene una responsabilidad específica y solo se comunica con la capa inmediatamente inferior.

Presentación

Interfaz de usuario (Web, Mobile, CLI). Equivale a la Vista.

Negocio (Service)

Donde reside la lógica compleja. Equivale al Controlador pero más robusto.

Acceso a Datos

Gestión de BD y persistencia. Equivale al Modelo/DAO.

Aislamiento Total

2. Arquitectura Hexagonal (Ports & Adapters)

Busca que el "Core" de la aplicación sea independiente de herramientas externas (BD, Frameworks, UI). La lógica de negocio está en el centro y se comunica mediante "Puertos".

[ Infraestructura ] -> [ Puerto (Interfaz) ] -> [ Lógica de Negocio (Core) ]
Ejemplo:
- Un Adaptador para MySQL.
- Un Adaptador para MongoDB.
* El Core no cambia, solo el Adaptador.
Escalabilidad Horizontal

3. Arquitectura de Microservicios

En lugar de una sola aplicación gigante (Monolito), dividimos el sistema en servicios pequeños e independientes que se comunican por red (REST/gRPC).

Independencia

Cada servicio puede usar su propia BD y lenguaje de programación.

Resiliencia

Si el servicio de "Pagos" falla, el de "Catálogo" sigue funcionando.

Reactividad

4. Arquitectura Orientada a Eventos (EDA)

Los componentes no se llaman entre sí directamente. En su lugar, emiten y reaccionan a "eventos" a través de un bus de mensajes (Broker).

1. El usuario compra -> Evento: "PEDIDO_CREADO"
2. El servicio de Stock escucha -> Actualiza inventario.
3. El servicio de Email escucha -> Envía confirmación.
* Todo sucede de forma asíncrona.

¿Cuál elegir?

  • 🚀 Pequeño/Mediano: MVC o Capas (Simplicidad).
  • 🏗️ Largo Plazo/Complejo: Hexagonal (Mantenibilidad).
  • 🌐 Gran Escala: Microservicios (Escalabilidad).
  • ⚡ Tiempo Real: Orientada a Eventos (Velocidad).
