# Día 1: Herencia Sellada - Control del Dominio 🏗️

## 🎯 Contexto Inicial
Hoy aprenderás a proteger la arquitectura de tu sistema. En la POO tradicional (Legacy), cualquier clase podía heredar de otra, lo cual es peligroso para el dominio de tu negocio. Hoy dominarás las **Sealed Classes** de Java 17 para decidir exactamente quién tiene permiso de extender tu código.

## 📚 Conceptos de Base
1.  **Sealed Classes (`sealed`, `permits`)**: Permiten al programador restringir qué clases pueden heredar de ella.
2.  **Seguridad Arquitectónica**: Esta característica es vital para crear APIs donde no quieres que alguien "inyecte" comportamientos extraños mediante herencia libre.
3.  **Subclases Finales o No-Selladas**: Las subclases de una sealed class deben declarar obligatoriamente si ellas mismas son `final`, `sealed` o `non-sealed`.

## 🛠️ Reto Práctico: Protegiendo el Ecosistema
1.  **Herencia Abierta**: Crea una clase abstracta `Persona` que permita ser heredada de forma tradicional.
2.  **Refactorización Sellada**: Refactoriza `Persona` a una **`sealed class`** que solo permita ser extendida por `Empleado` y una nueva clase `ConsultorExterno`.
3.  **Intento de Invasión**: Crea una clase `Desconocido` e intenta que herede de `Persona`. Mira el error del compilador.
